# ディレクトリ構造ルール

## プロジェクトレイアウト

```
cc-lmstudio-proxy/
├── src/                    # ソースコード
│   ├── index.ts           # エントリーポイント
│   ├── config.ts          # 設定読み込み
│   ├── proxy.ts           # プロキシサーバーロジック
│   ├── logging.ts         # ロギングユーティリティ
│   ├── env.ts             # 環境変数パース
│   ├── codec.ts           # エンコーディングユーティリティ
│   └── path.ts            # パス解決ユーティリティ
├── test/                   # テストファイル
│   ├── *.test.ts          # src 構造に合わせたテストファイル
│   └── helpers.ts         # テストユーティリティ
├── docs/                   # ドキュメント
│   └── rules/             # プロジェクトルールと規約
├── logs/                   # 実行時ログファイル（gitignore 対象）
├── package.json           # パッケージ設定
├── tsconfig.json          # TypeScript 設定
└── bun.lockb              # Bun ロックファイル
```

## ディレクトリ規約

### `src/`
アプリケーションのすべてのソースコードを含みます。各ファイルは以下を満たすべきです：
- 単一で明確な責任を持つ
- 他のモジュールに必要なものだけをエクスポートする
- すべてのエクスポートに型注釈を含む

### `test/`
すべてのテストファイルを含みます。構造は `src/` をミラーリングします：
- `src/foo.ts` → `test/foo.test.ts`
- 共有テストユーティリティは `test/helpers.ts` に配置

### `docs/`
プロジェクトドキュメントを含みます：
- `rules/` - アーキテクチャとコーディング規約
- その他必要なドキュメント

### `logs/`
実行時に生成されるログファイル。git にコミットしない。

## ファイル命名規則

### ソースファイル（`src/`）
- 小文字で `.ts` 拡張子
- 目的を反映する説明的な名前
- テストファイルは `.test.ts` サフィックス

### 設定ファイル
- `package.json` - Node.js パッケージメタデータ
- `tsconfig.json` - TypeScript コンパイラオプション
- `bun.lockb` - Bun 依存ロックファイル（バイナリ）

## インポートパスルール

1. **相対インポートのみ**: プロジェクト内のすべてのインポートは相対パスを使用
2. **`.ts` 拡張子を含む**: インポートでは常にファイル拡張子を含む
   ```typescript
   // 良い
   import { config } from "./config.ts";

   // 悪い
   import { config } from "./config";
   import { config } from "./config.js";
   ```

3. **ディレクトリインポートしない**: いつも特定のファイルをインポートする
   ```typescript
   // 良い
   import { foo } from "./utils/foo.ts";

   // 悪い
   import { foo } from "./utils";
   ```

## Git 構成

### 管理対象（コミットされる）
- `src/` - すべてのソースコード
- `test/` - すべてのテストファイル
- `docs/` - ドキュメント
- 設定ファイル（`package.json`, `tsconfig.json`）

### 非管理対象（gitignore）
- `logs/` - 実行時ログファイル
- `node_modules/` - 依存関係
- `*.log` - プロジェクト内のあらゆるログファイル

## 新しいディレクトリを追加する際

新しいディレクトリを作成する前に、以下を検討してください：
1. コードは既存のディレクトリに収められるか？
2. これは長期的な追加か、一時的な実験か？
3. 単一責任原則に従っているか？

新しいディレクトリを追加する際：
- 小文字の名前を使用する
- 必要でない限りネストしたディレクトリを避ける
- 構造が大きく変わる場合はこのドキュメントを更新する
